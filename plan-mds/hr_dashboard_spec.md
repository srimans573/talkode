# HR Dashboard Spec — Post-Session Analysis View

This is what HR sees AFTER a candidate's session is fully processed. The candidate
never sees any of this. This doc is the single source of truth for how that view
should read and what it must never do — read this before building the dashboard
component tomorrow.

---

## The core rule: SYNTHESIZE, never EVALUATE

This is the one rule everything else follows from. Two different things look similar
but are not:

- ✅ **Observation (synthesis)** — "they got stuck for 90 seconds, then self-corrected
  without a hint." True, verifiable, neutral. No judgment baked in.
- ❌ **Evaluation (judgment)** — "this is a strength" / "this is a weakness" / a score /
  a confidence level / "pro" / "con." Requires the system to decide what's good — which
  is exactly the open-ended, unfalsifiable judgment this whole project is built to avoid.

**"Clear" does not mean "evaluated."** A dashboard can be clean, confident, well-organized,
and narrative — and still never rate anything. That's the actual design target here.

If a judge asks "so is this candidate good?" — the answer is always:
*"We don't say. We show HR exactly what a senior engineer would have noticed, clearly
and with evidence — the judgment is still theirs, because it should be."*

### Explicitly banned from this view
- Any overall score or rating (1-10, %, letter grade, etc.)
- Any "confidence level" about the CANDIDATE (a confidence level about your own
  DETECTION quality, e.g. "high/medium confidence this was a real self-correction,"
  is fine — that's about the system's certainty, not a verdict on the person)
- "Pros" / "cons" / "strengths" / "weaknesses" labels
- A freely-generated "overall impression" paragraph not directly composed from
  detected events
- Ranking or comparing this candidate against others (unless real cross-session
  Redis data backs it — don't show a stat you can't prove)

## Structure: three layers, skim-to-deep

### Layer 1 — Session summary (top of page, no clicking required)
One short paragraph, composed FROM the actual detected event counts/types —
not freely generated by an LLM with no grounding. Template-filled, e.g.:

> "During this 14-minute session, the candidate asked one clarifying question
> early on, hit one stuck point that they resolved independently, and explained
> their reasoning before two major code changes."

Every clause must map to a real, clickable event below it.

### Layer 2 — Timeline (visual centerpiece)
A horizontal bar = session duration, with each detected moment as a colored
marker at its timestamp. Color-code by type (e.g. stuck = amber, self-correction
= green, question = blue, justification = purple). Should be legible at a glance,
even to a non-technical judge glancing at your screen.

### Layer 3 — Evidence on click
Click any marker → expands to show:
- Exact transcript snippet, quoted, timestamped
- Code diff at that moment (small/scoped — just the relevant change, not the
  whole file)
- The one-sentence factual label from the bounded LLM call
- If it was a Live Reactive Layer response: what the agent said back, shown here too

---

## Themed grouping (this is where "insight" comes from, without evaluation)

Group events by WHAT KIND OF THING HAPPENED, not by how good it was:

- **"Problem-Solving Process"** → stuck moments + self-corrections, told as a short
  narrative: "Spent ~90 seconds without code changes after starting the two-pointer
  approach, then said 'wait, this breaks on duplicates' and refactored the loop
  condition." Click → evidence.
- **"Communication While Coding"** → clarifying questions + architecture
  justifications, same treatment.
- **"Decision Points"** → explicit "I'll go with X over Y" moments — a short list
  of choices made and the stated reasoning, quoted.

This is synthesis: organizing real events into a clear shape. It is NOT evaluation:
nothing here says whether the shape is good. HR brings their own context (the role,
the team, the bar) to interpret it — that's intentional, not a gap.

## Build notes
- Read-only page. Given a session ID, one FastAPI GET route returns
  `session:{id}:meta` + `session:{id}:events` from Redis; render client-side.
- No real-time updates needed — the session is already finished by the time
  this is viewed.
- A simple landing page listing all sessions (`sessions:all` from Redis) lets
  you navigate to the right demo session at your table tomorrow.
- Spend real design/polish time here specifically — this is the screen judges
  will actually look at the longest. Clean color-coded markers + smooth
  expand/collapse on click matters more here than almost anywhere else in the build.
